# 15 — shadcn/ui + Tailwind v4 design system

**Depends on:** 04 (frontend shell exists). Can be implemented any time before 09 if the
run/results UI should be built on the component library from the start.
**Enables:** faster, consistent frontend work for plans 09, 13, and future polish.

## Goal

Install the full current shadcn/ui component set into `frontend/` and configure Tailwind
CSS using the version path supported by latest shadcn: Tailwind v4 with the
`@tailwindcss/vite` plugin, CSS-first theme tokens, and shadcn's generated component
source committed in the app. Configure the frontend so it can consume TypeScript and TSX
files even if the existing application code remains JavaScript during the first pass.

Official references to verify at implementation time:

- `https://ui.shadcn.com/docs/installation/vite`
- `https://ui.shadcn.com/docs/tailwind-v4`
- `https://tailwindcss.com/docs/upgrade-guide`

## Decisions

- Use the shadcn CLI as the source of truth: `npx shadcn@latest init`, then
  `npx shadcn@latest add --all`.
- Prefer the `new-york` style for new components, matching shadcn's current default.
- Use Tailwind v4, not a v3 compatibility setup. The Vite integration is
  `@tailwindcss/vite`, and `src/styles.css` should begin with `@import "tailwindcss";`.
- Keep components local to the frontend package under `src/components/ui/`, with shared
  helpers in `src/lib/`.
- Add the `@/*` import alias in Vite and TypeScript config so generated imports are
  stable.
- The current app is React 18 and JavaScript. Add TypeScript support in this plan even if
  the existing `.jsx` files are migrated incrementally. Before implementation, confirm
  whether the latest generated components still target React 18 cleanly. If shadcn's
  latest output requires React 19, upgrade deliberately and test Monaco, React Router, and
  Testing Library rather than mixing incompatible generated code.

## Deliverables

### 1. Tailwind v4 wiring

- Add runtime/dev dependencies required by shadcn and Tailwind v4:
  `tailwindcss`, `@tailwindcss/vite`, `class-variance-authority`, `clsx`,
  `tailwind-merge`, `lucide-react`, plus Radix and supporting packages generated by the
  selected shadcn components.
- Update `frontend/vite.config.js`:
  - add `tailwindcss()` to `plugins`;
  - preserve the `/api` dev proxy and Vitest config;
  - add the `@` alias to `frontend/src`.
- Replace the existing global stylesheet with Tailwind v4 imports and shadcn theme
  tokens, preserving any app-specific layout rules only when they are still needed.
- Remove obsolete Tailwind v3 files if the CLI or upgrade path creates/keeps them
  unnecessarily.

### 2. TypeScript readiness

- Add TypeScript dependencies and type packages:
  `typescript`, `@types/react`, `@types/react-dom`, and `@types/node`.
- Add Vite-compatible TypeScript config files:
  - `frontend/tsconfig.json`
  - `frontend/tsconfig.app.json`
  - `frontend/tsconfig.node.json`
- Configure `allowJs: true` and `checkJs: false` initially so the existing `.jsx` files
  keep working while new shadcn components can be generated as `.tsx`.
- Configure JSX with the React automatic runtime and include `src`.
- Add `baseUrl` and `paths` to both `tsconfig.json` and `tsconfig.app.json`:

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

- Update `package.json` scripts so TypeScript can be checked separately from the Vite
  build, for example:

```json
{
  "scripts": {
    "typecheck": "tsc -b --pretty false"
  }
}
```

- Do not require a whole-app JavaScript-to-TypeScript migration for this plan. Migrate
  existing files only when necessary to make generated components, imports, or tests
  compile cleanly.

### 3. shadcn initialization

- Create `frontend/components.json` using the current CLI schema.
- Configure aliases for:
  - `components`: `@/components`
  - `ui`: `@/components/ui`
  - `lib`: `@/lib`
  - `hooks`: `@/hooks`
- Add `src/lib/utils.ts` with the standard `cn()` helper.
- Prefer TypeScript output for generated shadcn files. Keeping app pages in `.jsx` is
  acceptable as long as Vite, Vitest, and `npm run typecheck` can resolve the TS/TSX
  components.

### 4. Install all shadcn components

Run the latest CLI from `frontend/`:

```sh
npx shadcn@latest add --all
```

Then audit the generated set against the component index in the official docs. The
expected set includes, at minimum: accordion, alert, alert-dialog, aspect-ratio, avatar,
badge, breadcrumb, button, button-group, calendar, card, carousel, chart, checkbox,
collapsible, combobox, command, context-menu, data-table, date-picker, dialog, drawer,
dropdown-menu, empty, field, hover-card, input, input-group, input-otp, item, kbd, label,
menubar, navigation-menu, pagination, popover, progress, radio-group, resizable,
scroll-area, select, separator, sheet, sidebar, skeleton, slider, sonner, spinner,
switch, table, tabs, textarea, toggle, toggle-group, tooltip, and typography.

Notes:

- Prefer `sonner` over the deprecated `toast` component unless the latest docs reverse
  that guidance.
- Some docs entries are patterns rather than single components. If `--all` skips a
  pattern such as data table, date picker, or combobox, add the underlying primitives and
  document any hand-created composition.
- Keep generated component source unmodified on first install except for project-required
  JavaScript/TypeScript adjustments. Local design changes should happen after the baseline
  build is green.

### 5. Integrate without redesigning the app

- Replace only obvious low-risk primitives first: buttons, badges, inputs, cards/sheets,
  tabs, separators, and skeleton states.
- Keep the problem browser and workspace behavior unchanged.
- Do not restyle Monaco editor internals in this plan.
- Add a small internal component gallery route or test fixture only if it materially helps
  verify that all installed components compile and render.

### 6. Documentation

- Add a short frontend design-system note, either in `README.md` or
  `docs/guide-frontend.md`, covering:
  - how to add/update shadcn components;
  - why Tailwind v4 is used;
  - how TypeScript is configured and when to migrate `.jsx` files;
  - where theme tokens live;
  - how to avoid editing generated components unless needed.
- Update `docs/00-overview.md` if the frontend stack changes from React 18 to React 19 or
  from JavaScript to TypeScript.

## Testing notes

- `npm run build` from `frontend/` must pass.
- `npm run typecheck` from `frontend/` must pass.
- `npm test` from `frontend/` must pass.
- Add or update Vitest coverage for any migrated app components.
- Manually smoke test:
  - login/signup;
  - problem list;
  - problem workspace;
  - editor loading and resizing;
  - light/dark theme behavior if enabled.
- Inspect the production bundle for accidental duplicate UI libraries or obsolete
  Tailwind v3 dependencies.

## Testable outcome

The frontend builds and tests with Tailwind v4 and the latest shadcn CLI output. Every
shadcn component available from the current official registry is present under
`frontend/src/components/ui/` or explicitly documented as a registry pattern, and the
existing Distry user flows still work unchanged.
